# Localization Implementation Plan — React Migration

## Objective

Migrate all hardcoded UI text from the React admin app to use `react-i18next`, consuming the existing `locale_resources-{culture}.json` files generated by the backend `ResourceBuilder`.

---

## 1. Backend Changes

### 1.1 Serve `Locale/` folder as static assets

**File:** `src/App/YOApp/Program.cs`

Add static file serving for the `Locale/` directory so JSON files are accessible at `/locale/locale_resources-{culture}.json`.

```csharp
app.UseStaticFiles(new StaticFileOptions
{
    FileProvider = new PhysicalFileProvider(Path.Combine(env.ContentRootPath, "App_Data", "Locale")),
    RequestPath = "/locale"
});
```

### 1.2 (Optional) New API endpoint for locale resources

If static file serving is insufficient, add:

```
GET /api/v1/localization/resources/{culture}
→ { "Dashboard.Widgets": "Widgets", ... }
```

**File:** `src/Modules/YandOne.Admin/API/LocalizationApiController.cs`

---

## 2. Frontend Infrastructure

### 2.1 Install packages

```
npm install i18next react-i18next i18next-http-backend i18next-browser-languagedetector
```

### 2.2 Create `src/i18n.ts`

- Backend: `i18next-http-backend` loading from `/locale/locale_resources-{{lng}}.json`
- Custom `parse` to flatten `[{Group, Locales: [{Name, Value}]}]` → `{ "Name": "Value" }`
- Language detection: read `ASP.NET_Culture` cookie (set by the `POST /localization/language` endpoint)
- Fallback: English hardcoded resource object so the app works even if no JSON file exists
- `debug: true` during development

### 2.3 Wire into `main.tsx`

Add `import './i18n'` at the top.

### 2.4 Language switcher component

Reuse existing `POST /api/v1/localization/language` endpoint from `localizationAPI.ts` to switch culture and update i18next.

## 3. Component Migration Strategy

### 3.1 Key naming convention

All keys follow the pattern:

```
{PageName}.{ElementDescription}
```

Examples:
- `UserManagement.Title` → "User Management"
- `UserManagement.AddNew` → "Add New User"
- `UserForm.FirstName` → "First Name"

### 3.2 Migration pattern per component

```tsx
// Before
<h3>User Management</h3>
<button>Add New User</button>

// After
import { useTranslation } from 'react-i18next';

const MyComponent = () => {
  const { t } = useTranslation();
  return (
    <>
      <h3>{t('UserManagement.Title')}</h3>
      <button>{t('UserManagement.AddNew')}</button>
    </>
  );
};
```

### 3.3 Fallback behavior

i18next falls back to the key itself when no translation exists. This means the app remains fully functional even before locale resources are configured in the DB — the keys serve as self-documenting English defaults.

### 3.4 Shared components (DataGrid, Buttons, etc.)

Shared components get their own key namespace:
- `DataGrid.Previous` → "Previous"
- `DataGrid.Next` → "Next"
- `DataGrid.Showing` → "Showing"
- `DataGrid.AddNew` → "+ Add New"
- `Form.Save` → "Save"
- `Form.Cancel` → "Cancel"
- `Form.Delete` → "Delete"
- `Form.Edit` → "Edit"

## 5. Implementation Order

1. **Phase 0**: Backend — serve `Locale/` as static files, install packages, create `i18n.ts`, wire `main.tsx`
2. **Phase 1**: Shared components (DataGrid, Form buttons, PageMeta) — highest reuse
3. **Phase 2**: Auth pages (SignIn, SignUp, Callback)
4. **Phase 3**: Settings pages (Basic, API, CSP, Optimization, File, Web, Caching)
5. **Phase 4**: CRUD pages (User, Role, Permission, SEO, Menu)
6. **Phase 5**: Provider pages (Email/SMS providers, Email templates)
7. **Phase 6**: OpenIddict pages (Client, Grant, Resources, Scopes)
8. **Phase 7**: AI Assistant pages (Assistant, Category, Theme, Prompt, LLM, Moderation)
9. **Phase 8**: Remaining pages (HTML Builder, Media, Task Builder, UI demos)
10. **Phase 9**: Navigation/sidebar links

---

## 6. Verification

After each phase:
- `npm run build` — TypeScript compilation must pass
- Check that all replaced text renders correctly with `t()` calls
